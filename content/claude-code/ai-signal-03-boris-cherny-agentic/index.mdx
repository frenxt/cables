# Signal from the Timeline #3: Boris Cherny on his actual Claude Code setup

*A recurring series: we pull a post from an AI leader, add live commentary from our production work, and ship it as a Cables entry.*

## The thread

> "I'm Boris and I created Claude Code. Lots of people have asked how I use Claude Code, so I wanted to show off my setup a bit. My setup might be surprisingly vanilla! Claude Code works great out of the box, so I personally don't customize it much."
>
> Key practices from the thread:
> - **"I use Opus 4.5 with thinking for everything"** — the heaviest, slowest model, always
> - Runs 5-10 parallel Claude instances across browser and terminal simultaneously
> - Uses iTerm2 notifications to manage work streams — one agent runs tests, another refactors, another drafts docs
> - **"/commit-push-pr"** — a custom slash command he runs dozens of times daily
> - CLAUDE.md: **"Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time"**
>
> — [**Boris Cherny (@bcherny)**](https://www.threads.com/@boris_cherny/post/DTBVlMIkpcm/im-boris-and-i-created-claude-code-lots-of-people-have-asked-how-i-use-claude) (creator of Claude Code, Anthropic), January 2026

## What we heard

Three things stand out from this thread:

**1. He chose the slowest model on purpose.** Cherny is not optimizing for latency — he's optimizing for output quality. On a coding task that might take 20 minutes of agent time, the difference between Haiku and Opus matters more than the difference between 90s and 150s wall-clock.

**2. Parallelism is the multiplier.** 5-10 simultaneous agents means he's not waiting on any single task. The notification system is the key pattern — you don't watch the agent, you get notified when it finishes and then review.

**3. CLAUDE.md is a running postmortem.** Every incorrect behaviour gets documented and fed back. The file is not set-and-forget configuration — it's a living record of what went wrong and why.

---

## What we actually do with this

We adopted all three patterns during the InterviewLM engagement.

**Model selection:** We run Opus for anything involving system design, agent orchestration logic, or multi-file refactors. We run Haiku for formatting, boilerplate, and test generation. The model routing decision is made at task classification time, not at the start of a session.

**Parallelism:** We run separate Claude Code sessions per subsystem — one for the agent persona logic, one for the evaluation harness, one for the API layer. This is the main reason we went from three weeks estimated to six days actual.

**CLAUDE.md as postmortem log:** Our convention: any time an agent misunderstands a project constraint, we add a `## Known constraints` bullet to CLAUDE.md within the same session. The file is reviewed at the start of every new engagement kick-off.

---

## The `/commit-push-pr` pattern

This is the highest-signal detail in the thread. Cherny runs this dozens of times per day. That cadence only makes sense if:

1. The command is instant (no friction)
2. The commit quality is already high enough to push without review
3. PRs are small — single-purpose, reviewable in 2 minutes

We implemented the equivalent in our own workflow. The slash command runs: lint → build → git commit with a generated message → push → open PR draft. The whole thing takes under 30 seconds. It made our average PR size drop from 400 lines to 80 lines.

---

## The one thing to steal from this

Add a `## Known constraints` section to your CLAUDE.md and write one bullet every time Claude misunderstands something in your project. After a week you will have a surprisingly complete mental model of where the model's defaults conflict with your codebase's assumptions.
