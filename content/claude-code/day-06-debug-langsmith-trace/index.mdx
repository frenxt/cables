# Day 6: Debug an agent from a LangSmith trace

When an agent fails, raw logs usually tell us *what* happened, not *why* it happened in that step sequence.

<WarStory title="The bug was in a branch we never looked at">
We spent hours tuning prompts before noticing the tool-call branch was never executed after a guard condition changed. The trace made it obvious in two minutes.
</WarStory>

## What we tried

We used a trace-review skill that forces a consistent walkthrough:

- identify the failing run and expected behavior
- isolate the first diverging step
- inspect tool inputs/outputs at that step
- classify root cause (prompt, tool contract, retrieval, guardrail, routing)

## What happened

Once we standardized that sequence, debugging stopped feeling like archaeology. We could move from symptom to concrete fix proposal quickly.

## What we learned

- Always compare a failing trace against a known-good trace.
- Treat routing decisions as first-class failure points.
- Save root-cause notes in the repo so future regressions are faster to triage.
